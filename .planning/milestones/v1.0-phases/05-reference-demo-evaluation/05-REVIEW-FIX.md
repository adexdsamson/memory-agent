## Fixed Issues

### CR-01: `run_eval` bypasses WAL checkpoint via private `_db.close()`

**Files modified:** `src/mnema/eval/baseline.py`
**Commit:** 9512bba
**Applied fix:** Replaced `await engine._t1._db.close()` with `await engine.t1.close()` in the `run_eval` finally block. The public `t1.close()` property runs `PRAGMA wal_checkpoint(FULL)` before disconnecting, ensuring in-flight WAL pages are flushed to the main database file. Committed together with IN-01 since both edits are in the same file.

---

### CR-02: `run_session` finally block leaks resources on consolidation failure

**Files modified:** `src/mnema/demo/coach.py`
**Commit:** 24890ed
**Applied fix:** Wrapped each of the three cleanup calls (`engine.consolidate()`, `engine.t1.close()`, `engine._scheduler.shutdown()`) in its own try/except block. `consolidate()` logs a warning on failure; the other two silently swallow exceptions. All three cleanups now run unconditionally even if an earlier one raises, preventing connection leaks and orphaned APScheduler threads.

### WR-02: `test_eval_baseline` allows naive to fail any number of probes

**Files modified:** `tests/test_eval_baseline.py`
**Commit:** 379b42c
**Applied fix:** Changed `assert results["probes_passed_naive"] <= 2` to `assert results["probes_passed_naive"] == 2`. The assertion message was updated to explain that naive must pass exactly 2 probes (protection + cross-session) and fail exactly 1 (supersession avoidance). This prevents a fully broken naive baseline (0 probes) from silently passing the gate. The eval still produces `probes_passed_naive == 2` (confirmed by the test run).
