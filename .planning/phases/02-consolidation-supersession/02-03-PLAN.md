<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → supersede() | old_id and user_id come from the consolidation pipeline core, not from external input in Phase 2 |
| t0_ref lookup → SQL | t0_ref is a string parameter bound via parameterized query — no interpolation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Tampering | supersede() — cross-user record retirement | mitigate | `AND user_id=?` predicate in the UPDATE statement. If old_id belongs to a different user, the UPDATE affects 0 rows and the transaction succeeds silently without retiring anything. Consolidation pipeline always passes the correct user_id from the staging queue item (D-02/D-03). |
| T-02-06 | Tampering | supersede() — partial write on exception | mitigate | try/except + rollback pattern; commit only on full success. aiosqlite single-connection WAL ensures rollback is atomic. |
| T-02-07 | Tampering | find_by_t0_ref() — returns superseded provisional | mitigate | `AND valid_until IS NULL` predicate excludes superseded records from idempotency lookup. |
| T-02-08 | Tampering | _record_params — graph_edges list mutation | mitigate | Caller (consolidation pipeline) passes a new list to new_record via model_copy; _record_params calls json.dumps on the list at call time. Per RESEARCH.md Pitfall 5: always pass a new list to model_copy, never mutate the existing graph_edges. |
</threat_model>
