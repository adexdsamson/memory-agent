        "EXTRACT_RECORDS:" → calls _extract(prompt), returns JSON string list of record dicts

        anything else → returns ""
    - _extract uses a keyword rule table: content containing "allerg"/"anaphyl"/"celiac"/"seizure"/
      "epilep"/"diabeti"/"medication"/"intolerant" → record_type="fact", protected=True, salience=1.0;
      content containing "prefer"/"love"/"like"/"enjoy"/"hate"/"dislike" → record_type="preference",
      protected=False, salience=0.7; default → record_type="preference", salience=0.5
    - _judge uses hashlib.sha256 of the concatenated (existing+new) content pair; hash mod 3 maps to
      index in ["distinct", "refine", "contradict"] — deterministic, never random
    - Each extraction returns a list[dict] with keys: content (str), record_type (str), salience (float),
      keywords (list[str]), summary (str); json.dumps of that list is the return value
    - StubLLM satisfies LLMProvider Protocol by STRUCTURAL SUBTYPING ONLY — no (LLMProvider) base class
    - Module docstring mirrors StubEmbedder: "No external deps, satisfies LLMProvider Protocol structurally"
    - version class attribute: "stub-v1"
    - From __future__ import annotations is first import after module docstring
    - Imports: hashlib, json (stdlib only — no math needed unlike StubEmbedder)
  </behavior>

    - Module docstring: "StubLLM — deterministic LLM provider for hermetic CI tests. Dispatches
      on sentinel strings (EXTRACT_RECORDS:, JUDGE_CONTRADICTION:) to produce deterministic JSON
      extraction results or contradiction verdicts. Satisfies LLMProvider Protocol structurally.
      No external dependencies — uses only stdlib hashlib and json."
    - from __future__ import annotations; import hashlib, json
    - No TYPE_CHECKING guard needed (no port imports at runtime)
    - class StubLLM: (no base class — D-08 structural subtyping)
      - version: str = "stub-v1" class attribute
      - __init__(self) -> None: no parameters (no dim like StubEmbedder needs)
      - async complete(self, prompt: str, *, model: str | None = None) -> str:
          dispatch on "EXTRACT_RECORDS:" → self._extract(prompt)
          dispatch on "JUDGE_CONTRADICTION:" → self._judge(prompt)
          default → return ""
      - def _extract(self, prompt: str) -> str:
          split on "EXTRACT_RECORDS:", strip the RHS as `content`
          apply keyword table (see behavior above) to produce list[dict]
          the keywords list in each extracted record can be derived by splitting
          content on whitespace and keeping words longer than 4 chars (simple heuristic)
          summary is content[:60].strip()
          return json.dumps([record_dict, ...])
      - def _judge(self, prompt: str) -> str:
          split on "JUDGE_CONTRADICTION:", hash the body with hashlib.sha256
          return ["distinct", "refine", "contradict"][int_hash % 3]

    NOTE: The sentinel-based design means the consolidation pipeline controls the prompt format.
    The StubLLM never inspects the semantic content for verdict logic — only the hash.
    This keeps test fixtures predictable: to force a specific verdict in a test, seed the
    existing+new content pair such that sha256(pair) % 3 equals the desired index.
    For tests that need a guaranteed "contradict" verdict, use content pairs whose hash mod 3 == 2.
    Compute the required pair in the test fixture setup (helper function in conftest or test file).
  </action>
